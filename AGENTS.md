# AGENTS.md

本文件定义本仓库中 agent 执行任务时需要遵守的项目级约束。目标是保证后续所有实现都围绕“人员任职资格评价 APP”这一业务目标持续演进，而不是偏离到通用后台模板。

## 1. 项目目标

本项目用于建设一套人员任职资格评价系统，支持：

- 管理员配置评价批次、等级、维度、分类、评分项
- 评委对人员逐项评分
- 系统基于加权得分率自动定级
- 系统对批次内人员能力分布进行统计分析

当前仓库实现的是可运行 MVP，目标是先验证业务规则与交互流程，再逐步演进为更完整的数据库驱动正式系统。

## 2. 当前技术形态

- 后端：原生 Node.js HTTP 服务
- 前端：原生 HTML / CSS / JavaScript
- 数据存储：SQLite 数据库，默认文件为 `data/grade-evaluation.db`
- 种子数据：`data/store.json`
- 测试：Node 内置 `node --test`
- 安全基线：密码必须以哈希形式存储，不再接受新增明文存储方案

不要在没有明确理由的情况下随意替换整个技术栈。若引入框架或数据库，需要保留现有业务规则和数据结构语义。

## 3. 核心业务规则

这些规则默认视为系统基线，未经明确需求不要擅自修改：

### 3.1 评分规则

- 单项评分值固定为：`0 / 1 / 3`
- 每个评分项可配置权重
- 当前默认权重档位为：`1.0 / 1.5 / 2.0`
- 单项加权得分 = `评分值 × 权重`
- 单项满分 = `3 × 权重`

### 3.2 定级规则

- 综合评价依据 `加权得分率`
- 加权得分率 = `实际加权得分总和 / 满分加权得分总和`
- 等级根据得分率区间自动判定
- 高等级可配置关键项门槛
- 关键项门槛包括：
  - 关键项最低得分率
  - 关键项不得出现 0 分

### 3.3 批次与体系规则

- 评价按批次进行
- 同一批次内，一个人员仅允许存在一份最终评价记录
- 草稿批次允许编辑评价体系
- 激活批次后，评价体系冻结，不应再被直接修改
- 历史批次结果必须与其评价体系快照一致

### 3.4 统计规则

当前版本至少支持：

- 等级分布
- 维度平均得分率
- 个人结果列表

## 4. 目录职责

### `src/`

放后端代码与核心业务逻辑。

- `server.js`：HTTP 服务与路由
- `store.js`：数据访问与业务操作
- `scoring.js`：评分计算、定级、统计聚合

### `public/`

放前端静态资源。

- `index.html`：应用入口
- `app.js`：前端交互逻辑
- `styles.css`：页面样式

### `data/`

放数据库文件与种子数据。

- `grade-evaluation.db`：运行时数据库
- `store.json`：首启导入种子数据

### `docs/`

放需求、架构、接口、数据库设计和路线图文档。若业务规则发生变化，必须同步更新相应文档。

### `test/`

放自动化测试，当前重点覆盖评分和定级规则。

## 5. 开发约束

### 5.1 修改评分逻辑时

若修改以下任一内容，必须同步补充或更新测试：

- 评分档位
- 权重规则
- 关键项门槛
- 等级判定逻辑
- 统计聚合逻辑

重点测试文件是：

- `test/scoring.test.js`

### 5.2 修改数据结构时

若修改以下任一内容，必须同步更新文档：

- 批次模型
- 评价体系模型
- 评分项字段
- 结果字段
- API 返回结构

至少检查这些文档是否需要更新：

- `docs/requirements.md`
- `docs/architecture.md`
- `docs/api.md`
- `docs/database-design.md`

### 5.3 前端改动要求

- 保持 PC 和移动端浏览器可用
- 不要为了堆功能引入明显复杂但无必要的 UI 框架
- 配置页、评分页、统计页是核心路径，优先保证这三类页面可用性

### 5.4 后端改动要求

- 新增接口时保持命名与现有 REST 风格一致
- 业务计算优先复用 `src/scoring.js`
- 不要把评分算法散落到路由层和前端层
- 输入校验失败应返回明确的 4xx 错误，不要统一吞成 200 或模糊报错
- 列表查询优先支持分页、筛选、排序，避免继续扩大 `app-state` 返回体

## 6. 后续演进优先级

默认优先级如下：

1. 增加密码哈希和更稳妥的鉴权方案
2. 支持批量导入人员和评价体系
3. 增加部门/岗位维度统计
4. 支持多人评委独立评分后汇总
5. 接入企业单点登录与审计日志

## 7. 不应轻易改变的原则

- 不要把“加权评分 + 自动定级”简化回纯等权评分
- 不要去掉批次冻结机制
- 不要把高等级关键项门槛直接删除，除非产品需求明确变化
- 不要在没有文档同步的情况下改动核心规则
